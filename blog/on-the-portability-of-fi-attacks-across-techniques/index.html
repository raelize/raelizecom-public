On the portability of FI attacks

(across techniques)

Wednesday, Jul 16, 2025

We started analyzing the security of Google’s Nest Wifi Pro at the end of 2023. Of course, besides overcoming several physical practicalities (i.e., dumping the flash), this involves a lot of reverse engineering of software. As you may have already realized, at Raelize, we like to look beyond just software vulnerabilities. Therefore, while reverse engineering the software stack, we decided to test its resilience to electromagnetic (EM) glitches attacks as well.

Please note, we actually discovered several software vulnerabilities, mostly in low-level software running at a high privilege, however, none of these vulnerabilities appeared yet in Google’s or Qualcomm’s monthly security bulletins. Hence, we cannot discuss them yet at this time. Nevertheless, we also identified a hardware vulnerability that we successfully exploited using EM glitches. This attack was presented at hardwear.io USA 2025 and will be described in more detail in a series of blog posts.

  • Google Wifi Pro: Glitching from Root to EL3 - Part 1 - Characterization
  • Google Wifi Pro: Glitching from Root to EL3 - Part 2 - Arbitrary read/write at EL3

In this first blog post, we will explain how we characterized Qualcomm’s IPQ5018 SoC, in order to determine its susceptibility towards EM glitches.

We have organized the content in the following sections:

Background

Google’s Nest Wifi Pro is a consumer WiFi router available in many areas in the world. We identified and exploited the vulnerability on a device running firmware version 3.73.406133, which was the latest available version at the time.

As far as we know, the vulnerability cannot be patched, as it’s a hardware vulnerability, and the code construction that we leveraged for the exploitation remained unchanged. Therefore, the attack described in this blog post should apply to devices running the latest firmware version.

Note, we were able to perform this characterization test, which relies on running arbitrary code in Android, by leveraging a Secure Boot bypass that we disclosed to Google. This vulnerability gave us a root shell, and among a few other things, the power to modify the init.rc file, which is used to configure Android at early boot. This allowed us to mask the setting of /proc/sys/kernel/modules_disabled to 1 in order to load Linux kernel modules at runtime.

LKMs at runtime. If this Linux Kernel security feature is set to 1, kernel module loading is disabled until the system is rebooted.

Unfortunately, this Secure Boot vulnerability is still under embargo, hence, we cannot discuss it publicly (yet). For this blog post, it’s sufficient to raelize that this vulnerability gave us a root shell and the power to load custom kernel modules. The vulnerability we identified is very similar to the vulnerability (i.e., CVE-2024-22013) that Sergei discussed in 2024 in his presentation at Hardwear.io Netherlands 2024.

Characterization

Whenever it’s possible, we like the characterize the susceptibility of a target towards glitches before commencing to the real attack. This allows us, in a controlled and favorable setting, to determine if the underlying platform is vulnerable to glitches. In this case, this characterization step, towards a successful attack, allowed us to identify that Qualcomm’s IPQ5018 SoC is vulnerable to EM glitches.

This vulnerability, as it is a hardware vulnerability, cannot be fixed. You could, even though it’s a bit of a stretch, compare characterization to the identification of a software vulnerability, after which you need to demonstrate its impact by exploiting it.

Tooling

Keysight’s EM tooling works very similar to all EMFI tooling. A (very) high voltage is driven through a coil in order to generate an electromagnetic field. This allows us to introduce faults at the transistor level due to eddy currents within the chip’s circuitry. As a result, and this is just one example, it allows us to affect the operation of the CPU in order to corrupt the instructions that are executed. We have been stressing this again and again, if you consider only the skipping of instruction, you are not exploring the full potential of FI attacks. Hence, we always think and reason along the lines of corrupting instructions (i.e., changing opcodes and operands) when discussing FI attacks on the CPU of a chip.

We use this serial interface to communicate with the test code that we used for the characterization.

Trigger

Being able to time the glitch is fundamental for FI attacks. We decided on this device to leverage the GPIO signal used for the factory reset button. We simply soldered a wire to one of the reset button pads in order to access the GPIO signal at our best convenience.

This GPIO is configured in the init.rc at boot as is shown below.

on init
    Export FDR button GPIO
    write /sys/class/gpio/export "487"
    write /sys/class/gpio/gpio487/active_low "1"

As the GPIO signal is configured as an input pin, we re-configure it as an output pin, by modifying the init.rc. This prevents us from performing a factory reset, but that is likely not something we want to do anyways.

on init
    # Export FDR button GPIO
    # write /sys/class/gpio/export "487"
    # write /sys/class/gpio/gpio487/active_low "1"

    write /sys/class/gpio/export "487"
    write /sys/class/gpio/gpio487/active_low "1"
    write /sys/class/gpio/gpio487/direction "out"
    write /sys/class/gpio/gpio487/value "3"

We confirmed that we can control this GPIO pin using devmem from our root shell.

We can set the GPIO high using (i.e., trigger up. This allows us to identify, using an oscilloscope, when the relevant part of our test code start executing.

/ # devmem 0x01016004 32 0x3

We can set the GPIO low using (i.e., trigger down). This allows us to identify, using an oscilloscope, when the test code has finished executing.

/ # devmem 0x01016004 32 0x0

Now that the address and the values to control the GPIO are known, we can also do this also directly from code (i.e., our test code).

Reset

When glitches are injected, it’s highly likely the target may, at some point, enter into an erroneous state. Therefore, a reliable reset procedure in order to restore the target to a known state and flawlessly continue with the experiments. We decided to directly control the external power supply in order to perform a power-on reset whenever we need to reboot the target.

We use a solid state relay, driven by a GPIO pin from our glitcher, in order to switch the original power supply on and off.

Characterization code

The Linux kernel module that we used for the characterization implements the code that shown below.

#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/types.h>
#include <linux/dma-mapping.h>
#include <asm/cacheflush.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Raelize");
MODULE_DESCRIPTION("CHARACTERIZATION 1 for Google's Nest Wi-Fi Pro (3.73.406133)");
MODULE_VERSION("1.0");

static u32 _command = 0x1;
static u32 _iterations = 0x0;

module_param(_command, uint, S_IRUGO);
module_param(_iterations, uint, S_IRUGO);

#define o "add r7, r7, #1;"
#define t o o o o o o o o o o
#define h t t t t t t t t t t
#define d h h h h h h h h h h
#define x d d d d d d d d d d

static int unrolled_loop(volatile u32 *trigger) {
    volatile u32 counter = 0;

    *trigger = 0x3;

    asm volatile(
        "mov r7, #0;"
        x
        "mov %[counter], r7;"

        : [counter] "=r" (counter)
        :
        : "r7", "r12"
    );

    *trigger = 0x0;

    printk(KERN_ALERT "AAAA%08xBBBB%08xCCCC\n", counter, counter);
    return 0;
}

static int __init characterize_1_init(void) {
    volatile uint32_t *trigger;

    printk(KERN_ALERT "characterize_1 (init)!\n");

    trigger = ioremap(0x01016004, 4);

    if(_command == 1) {
        unrolled_loop(trigger);
    } else {
        printk(KERN_ALERT "characterize_1 (command (%d) unsupported)!\n", _command);
    }
    return 0;
}

static void __exit characterize_1_exit(void) {
    printk(KERN_ALERT "characterize_1 (exit)!\n");
}

module_init(characterize_1_init);
module_exit(characterize_1_exit);

The function characterize_1_init is executed directly after the Linux kernel module is loaded using insmod. The module parameter _command, which is passed when the kernel module is loaded, is used to trigger the execution of the unrolled_loop function. This function implements the following steps:

  1. Set trigger high by writing 3 to 0x01016004
  2. Set register r7 to 0
  3. Increase the r7 with 1 for 10,000 times
  4. Set trigger low by writing 0 to 0x01016004
  5. Print the value contained in r7 in the kernel log

To determine if Qualcomm’s IPQ5018 SoC is sensitive to EM glitches, we inject the glitch while the counter is being increased using the add instructions. When the glitch results in a fault, that alters the intended behavior of an add instruction, we should see that effect reflected in a counter value that’s different than the expected value (i.e., 10,000).

We can run the kernel module as follows:

# insmod characterize_1.ko _command=1 _iterations=10000 && rmmod characterize_1
[ 1054.149388] characterize_1 (init)!
[ 1054.149491] AAAA00002710BBBB00002710CCCC
[ 1054.176611] characterize_1 (exit)!

As you can see, the counter value is set to 0x2710, which is 10,000 in hex. Printing the counter value twice allows us to identify responses where the counter has been modified by a communication malfunction. This assures that we do not classify experiments incorrectly.

Characterization code timing

We measured the time between the trigger up and down moment using an oscilloscope in order to get an idea of when to inject the glitch.

Based on this timing, we decided to inject our glitch between 20,000 us and 40,000 us after the trigger goes up. This guarantees that we are only injecting glitches when the add instructions are executed and we are hitting no surrounding code. To determine if Qualcomm’s IPQ5018 SoC is vulnerable, it does not matter too much which add instruction we corrupt, as long as we corrupt one.

Characterization code boot time

The duration of one experiment is the biggest disadvantages for characterizing using a kernel module. The kernel module can only be loaded after having booted into the root shell, which take a significant amount of time, especially compared to implementing the test code directly in the bootloader (e.g., U-Boot). We expect to reset the target often, so the overhead to boot, is significant.

We decided to speed up the boot process in order to load a kernel module earlier after boot. We made the following additions to init.rc at the end of on init, which spawns our root shell very early at boot. This results in a non-functional device, however, it allows us to load a kernel module.

on init
    ...
    write /sys/devices/system/cpu/cpu0/online "1"
    write /sys/devices/system/cpu/cpu1/online "0"
    exec /factory/raelize/bin/sh

The /factory/raelize/bin/sh is a symlink to a pre-compiled busybox binary that we put on the device as well. We also made sure that only core 0 (i.e., cpu0) is available at the moment of testing. This should prevent any possible issues due to scheduling or parallel execution.

Characterization results

We scanned the surface of Qualcomm’s IPQ5018 SoC in a grid of 10 x 10 with Keysight’s EMFI probe. After performing roughly 45 experiments at each location, where we randomize the power of the EM probe between 10% and 100%, we observe several locations where the counter value is different than the expected value.

We plot the experiments using our TAoFI-Analyzer, for which the results are shown below. The red dots represent the locations where we are able to modify the counter value.

Several of the interesting results are listed below.

Response Description
AAAA00002710BBBB00002710CCCC expected (i.e., glitch has no impact)
AAAA0000270fBBBB0000270fCCCC counter - 1
AAAA00002790BBBB00002790CCCC counter + 0x80
AAAA000027c0BBBB000027c0CCCC counter + 0xb0
AAAA4000198eBBBB4000198eCCCC Address in DDR
AAAA6fb91dacBBBB6fb91dacCCCC no idea

It’s difficult to determine what type of fault exactly materialized due to the EM glitch we injected. However, seeing a counter value of 0x270f (i.e., counter - 1) is typically a sign that the target is indeed vulnerable. Moreover, seeing DDR addresses end up in the counter is a good indication for instruction corruption as a value from another register (likely) got moved into the register used for the counter (i.e., we corrupted the operand).

To increase our chances for success, we decided to fix the EM probe on a location where we were able to change the counter value to 0x270f (i.e., counter - 1). Moving the probe across the surface of the Qualcomm’s IPQ5018 SoC is a time consuming process. Therefore, by fixing the probe, we can search the glitch parameter search space (i.e., location, timing and power) more efficiently.

Conclusion

We successfully determined Qualcomm’s IPQ5018 SoC is susceptible to EM glitches. We determined this by running characterization code in a custom Linux kernel module. Identifying a vulnerable location is fairly simple and can be done within hours.

The results of the characterization test gave us good confidence we can alter the intended behavior of the instructions executed by the CPU, including the Secure Monitor code. We decided to move forward with performing an attack on the Secure Monitor. This adventure is described in more detail in part 2 in this series of blog posts.
