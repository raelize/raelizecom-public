Introduction

In 2025 we presented at hardwear.io in the Netherlands an attack on Google’s TV Streamer 4K where we used an ElectroMagnetic (EM) glitch to escalate privileges from shell to root from an adb shell. Back then, we did not rely on Claude, but for this blog post, we asked Claude to reproduce the attack. We humans created the physical setup, but Claude took care of the entire software stack. We just pointed it to our presentation and the different relevant APIs that it should use. We shared no code with Claude that we made during our original research.

Target

Google’s TV Streamer 4K is designed around Mediatek’s MT8696 System-on-Chip (SoC). This is a fairly recent platform with all the bells and whistles that you expect from a modern SoC. It includes a quad-core CPU with ARM’s Cortex-A55 architecture (i.e., it supports both ARM’s AArch32 ISA and AArch64 ISA).

Android Debug Bridge (ADB)

The ADB feature is not enabled on the device by default. It’s something a developer would enable in order to view debug logs, install apps, runs shell commands, transfer files, etc. Hence, it’s a useful entry point for an attacker as well, assuming the user of the target is the attacker, which often holds up for these type of devices. Google is of course aware of this, hence, the adb shell is heavily restricted, and having it enabled should not compromise most of the assets. All of this is, to a certain extend, very similar on all Android phones, including Google’s Pixel devices.

The ADB feature is enabled in Android’s Developer Options, which are enabled by pressing the Build Options seven (7) times in the menu using the remote. Once enabled, the ADB feature exposes itself via USB and the user can get a shell with standard tooling.

We can get some info:

niek@laptop:~$ adb shell getprop ro.build.fingerprint
google/kirkwood/kirkwood:14/UTTK.250729.004/14066481:user/release-keys

We can run custom code, as the shell user:

niek@laptop:~$ $CC program.c -o program
niek@laptop:~$ adb push program /data/local/tmp
niek@laptop:~$ adb shell /data/local/tmp/program
I'm a program running as the shell user!

We can get the kernel configuration:

niek@laptop:~$ adb pull /proc/config.gz
/proc/config.gz: 1 file pulled, 0 skipped. 8.4 MB/s (41604 bytes in 0.005s)

The kernel configuration indicated to us that the security posture, at least for the Android Kernel, is very similar as a Pixel 9 mobile phone. This means, it should not be too easy to compromise its security.

Setup

Claude does not do any (physical) wiring yet. Maybe we should invest in a humanoid to do that work also for us? The setup is fairly simple for a EMFI setup due to the restrictions imposed by the target. A single USB-C cable is used for powering the target and communication. As the adb shell that we use is restricted, we are unable to set a trigger for timing, hence we did not add any logic to consume a trigger. We use a YEPKIT YKUSH3 to be able to reset the target from code.
